# 《110实验室前沿技术分组研究中期报告》修改建议来源总结

本文档用于解释：你刚刚收到的那批“问题与修改建议”是从哪里来的，为什么会集中指向封面、模型对比、指标补充、演示主流程、问题表更新和关键文件路径等位置。

结论先说：

- 这些建议**不是单一来源**
- 它们主要来自 **3 类依据**
  1. `110实验室关于开展前沿技术分组研究与专题汇报的通知 (3).pdf` 里的明确要求
  2. 你当前项目仓库中的**最新实验与交付状态**
  3. 面向中期答辩/老师阅读时常见的**表达与结构化要求**

---

## 1. 第一类来源：实验室通知里的“硬性要求”

根据通知 PDF，老师/组织方对这类分组研究报告与汇报的要求非常明确，核心关键词包括：

- 研究闭环
- 实验结果
- 真实系统 / 真实实验 / 真实结果
- 演示与汇报
- 代码与材料归档
- 团队分工
- 结果可复现

### 1.1 通知中直接对应到你这批建议的内容

通知中可以提炼出几条非常关键的要求：

1. **要完成从选题调研到实验实现再到成果汇报的完整过程**
2. **结题文档应包含**：课题简介、方法设计、实现过程、实验设置、结果分析、问题总结、后续改进方向、参考文献
3. **汇报 PPT 应重点体现**：研究问题、方法思路、系统设计、实验结果、结论与思考
4. **需要有应用演示或实验演示**
5. **需要代码与材料归档，并确保成果可复现、可查阅**
6. **汇报不能停留在概念介绍层面，应尽量展示真实系统、真实实验、真实结果**
7. **评价维度里明确包含**：
   - 选题价值
   - 完成质量
   - 技术深度
   - 汇报表达
   - 团队协作

### 1.2 因此，哪些建议是直接被通知“推出来”的

下面这些建议，基本都能直接在通知要求里找到依据：

- **封面和成员分工不能留占位符**
  - 对应通知中的“团队协作”“任务分工”“成果提交规范”
- **要补全演示主流程**
  - 对应通知中的“应用演示或实验演示 1 份”
- **要把脚本说明升级成可展示的完整流程**
  - 对应通知中的“避免停留在概念介绍层面”
- **要补代码、结果、材料归档入口**
  - 对应通知中的“代码与材料归档”“成果可复现”
- **要强调真实实验、真实结果，而不是只讲方法**
  - 对应通知中的“真实系统、真实实验、真实结果”
- **问题与后续改进部分要写实，不能停留在空泛表述**
  - 对应通知中的“问题总结、后续改进方向”

换句话说，这一部分建议的本质是：

> 报告要从“写过一些内容”升级成“满足实验室中期汇报和答辩要求的正式材料”。

## 2. 第二类来源：你的项目已经更新了，但报告没完全跟上

这类建议不是来自通知本身，而是来自**“报告文本版本”落后于“你代码仓库和实验结果版本”**。

你当前项目里已经有很多新内容：

- 正式模型对比表
- XGBoost 基线
- 动量基线
- `NDCG@K` / `HitRate@K`
- 市场阶段分析
- Demo 流程图、关键命令页、结果文件说明
- 冻结提交链路与 README 整理

而你贴出来的那批建议里，很多句子都明显在做“版本同步”工作。

### 2.1 明显属于“文档没跟上仓库”的建议

- “XGBoost 暂未纳入正式结果”要改
- “继续补动量基线”要改
- `4.3 评价指标` 需要补 `NDCG@5/10/20` 和 `HitRate@5/10/20`
- `5.1 模型总体对比` 要升级成正式模型对比表
- `5.2 Walk-Forward` 后面要接市场阶段分析
- `10.2 关键文件清单` 中 `app/result.csv` 要改成 `app/output/result.csv`
- 要补正式对比表、市场阶段分析表、Demo 主流程表、冻结提交材料说明

这类建议的真正来源不是“老师额外加要求”，而是：

> 你的项目已经迭代到了更完整的状态，但中期报告正文还停留在较早版本。

因此，这部分建议本质上是在做：

- 文档与代码同步
- 文档与结果同步
- 文档与最新答辩口径同步

## 3. 第三类来源：答辩叙事和老师阅读体验

还有一类建议，并不是通知里逐条写出来的，也不是单纯因为仓库更新了，而是来自**“什么样的中期报告更好读、更像正式汇报材料”**。

### 3.1 这类建议的典型特征

- 图号顺序要统一
- 回测结论要写得更直接
- 系统实现部分不要只是脚本清单
- 演示部分要按主流程组织
- 问题表要写成“问题-当前表现-已采取措施-后续方案”
- 解释逻辑要更完整，比如：
  - 为什么 `LSTM sl20` 是正式默认方案
  - 为什么动量基线命中率高但收益转化不如主线
  - 为什么稳定性问题和市场状态有关

### 3.2 这些建议为什么会出现

因为老师在看中期报告时，通常不只是检查“你有没有做”，还会隐含看这几件事：

1. 你有没有形成清晰主线
2. 你能不能解释“为什么这么选”
3. 你能不能把结果组织成一个可讲的故事
4. 你是不是已经具备答辩材料雏形

所以才会出现这类建议：

- “把 `7.2` 改成完整 Demo 主流程”
- “把 `6.3` 的回测结论写得更明确”
- “把 `5.2` 的现象描述升级成结构化阶段分析”
- “补一页关键命令截图”
- “图号别乱，别让老师感觉是后拼进去的”

这类建议本质上是在做：

> 从“项目记录文档”升级到“可用于中期答辩和老师快速理解的汇报文档”。

## 4. 具体到你贴出的建议，分别大概属于哪一类

### 4.1 主要来自通知硬性要求

- 封面成员信息去占位符
- `9.2 成员分工` 写具体
- `7.2 可演示主流程` 做成能展示的流程
- 强调系统、实验、结果、演示、归档

### 4.2 主要来自项目版本更新

- 补 XGBoost、动量基线
- 补 `NDCG/HitRate`
- 升级 `5.1 模型总体对比`
- 接入市场阶段分析
- 修正 `app/output/result.csv` 路径
- 补充正式模型对比、冻结提交、Demo 文档入口

### 4.3 主要来自答辩表达优化

- 图号统一重排
- 回测结论写得更直接
- 问题表改成“当前表现-已采取措施-后续方案”
- 用更清楚的话解释“为什么选 `LSTM sl20`”
- 用“主流程化”方式写系统展示章节
