当前程序的工程能力、风控工具、回测体系、可复现性明显强于压缩包参考程序；但当前正式输出的单切片收益率低于压缩包自带输出，也低于压缩包记录的最好分数。

## 3. 收益率复算结果

本次按压缩包 `test/score_self.py` 的同口径复算：

1. 提交结果最多包含 5 只股票。
2. 对每只股票，取测试集中最后 5 条记录。
3. 个股收益率 = 最后一条记录开盘价 / 第一条记录开盘价 - 1。
4. 最终得分 = 个股收益率与提交权重的加权和。

复算结果如下：

| 项目 | 得分 |
|---|---:|
| 当前 `app/output/result.csv` | `-0.005102` |
| 压缩包 `output/result.csv` | `0.025179` |
| 压缩包 `model/60_158+39/final_score.txt` 记录最好分数 | `0.037838` |

结论：

- 当前输出低于压缩包当前可见输出，差值约 `-0.030281`。
- 当前输出低于压缩包记录最好分数，差值约 `-0.042940`。
- 因此，当前正式输出不能说超过压缩包程序。

## 6. 稳定性对比

当前程序有更完整的稳定性和回测体系。当前默认回测指标如下：

| 指标 | 当前程序 |
|---|---:|
| 成本后累计收益 | `1.171246` |
| 成本后 Sharpe | `4.019488` |
| 成本后最大回撤 | `-0.090067` |
| 平均换手率 | `0.956671` |
| 总交易成本 | `0.057400` |
| walk-forward RankIC 均值 | `0.027982` |
| walk-forward Top5 平均收益均值 | `0.007502` |

解读：

- 当前程序在多期回测中的组合收益很强。
- 成本后 Sharpe 较高，说明组合层有效。
- 最大回撤可接受，但不是低回撤策略。
- 平均换手率接近 `1.0`，说明每期几乎满换仓，真实执行会比较依赖交易成本和滑点假设。
- RankIC 均值只有 `0.027982`，说明模型排序信号本身不够强。
- 三个 walk-forward fold 里有两个 RankIC 为负或接近 0，收益更多来自组合筛选、风险调整和少数强票，而不是模型分数的稳定单调性。

## 9. 当前程序主要不足

### 9.1 当前正式输出收益率不理想

当前输出在同口径测试切片上为 `-0.005102`，低于压缩包当前输出 `0.025179`。

直接原因：

- `600115` 贡献为 `-0.007762`，拖累明显。
- `600584` 也为负贡献。
- 权重和只有 `0.9`。

### 9.2 排序信号不够稳定

当前 walk-forward `rank_ic_mean=0.027982`，并且部分 fold 的 RankIC 为负或接近 0。说明模型对全截面排序的稳定性不足，收益更多依赖组合筛选与局部强票。

### 9.3 换手率偏高

平均换手率约 `0.956671`，每期几乎满换仓。比赛回测里可接受，但如果考虑实盘执行，会明显受交易成本、滑点和容量限制影响。

### 9.4 风控和比赛得分存在取舍

当前方案使用单票权重上限 `0.18`，导致权重和为 `0.9`。这对风险控制有帮助，但在比赛单切片计分中可能吃亏。

### 9.5 文档中存在旧结论

部分旧文档曾写到当前结果超过压缩包当前输出，但最新复算显示该结论已经过期。应以本文件中的复算结果为准。

## 11. 最终判断

如果目标是“当前提交文件谁收益高”，压缩包更好。

当前程序 `app/output/result.csv` 在同口径测试切片上的得分是 `-0.005102`，压缩包自带输出是 `0.025179`，压缩包记录最好分数是 `0.037838`。

如果目标是“谁的工程体系更完整、更适合继续优化”，当前程序更好。

当前程序已经具备完整的训练、推理、回测、稳定性分析、风控、消融实验、提交校验和冻结机制。它的问题不是框架弱，而是当前这版正式输出没有选出适合该测试切片的股票，并且排序信号、换手率、单票风险仍需要进一步优化。

更合理的后续路线是：保留当前工程框架，把压缩包的 Transformer 长序列和跨股票注意力吸收到当前系统中，做 LSTM + Transformer + 规则风控的融合，而不是简单用某一边完全替换另一边。
